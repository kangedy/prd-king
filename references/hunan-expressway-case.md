## 政策对齐的关键变更模式

当集团发布正式管理办法后，PRD需对齐更新。以下是典型的变更维度参考：

| 参数维度 | 旧版典型值 | 新版典型值 | 变更类型 |
|:---------|:----------|:----------|:--------|
| 会员等级 | 多级免费体系 | 少级免费 + 付费卡 | 结构重构 |
| 免费等级条件 | 消费额分段 | 注册即享 + 累计积分达标 | 规则变更 |
| 积分获取 | 消费+活动+行为 | 基础消费+等级加成+任务 | 范围扩展 |
| 积分使用范围 | 全场景直接抵扣 | **部分场景限制** | 政策限制 |
| 单笔抵扣上限 | 无限制 | ≤50%（可配置） | 新增限制 |
| 积分有效期 | 固定月数 | 固定天数，FIFO | 规则变更 |
| 清算主体 | 多主体净额结算 | **单一平台公司** | 结构重构 |
| 成本分担 | 未明确 | 平台+商户按比例 | 新增规则 |
| 优惠券 | 少数基础类型 | 多场景×多维+风控 | 大幅扩展 |
| 风险管理 | 无 | 运营/技术/法律三维 | 全新章节 |

## 增量修订案例（重点）

**场景**：集团发布正式管理办法后，PRD需对齐新政策。

**错误做法**：读取新旧文档→生成全新文档→丢失50%+原始内容（功能点表格、流程图全部丢失）。

**正确做法**（增量修订4步）：
1. 索引旧版（数百段落、数十表格）
2. python-docx精准替换：版本号、等级定义、积分规则、价格
3. lxml结构插入：新结算模块、风险防控章节、附录
4. 表级追加：对外接口、验收标准

**结果**：文件大小略增，保留全部原始内容+流程图+所有功能点表格。

详见 `references/prd-incremental-revision.md` 完整技术方案。

## PRD对齐政策文档的标准工作流

当甲方发布正式管理办法后，PRD需要对齐更新时：

1. **完整提取两份文档**：同时读取新政策文档和现有PRD
2. **编制差异对比表**：逐章节对比，标注变更性质（重构/修正/新增）
3. **识别需变更的模块**：将差异映射到PRD中的具体模块ID
4. **编号新模块**：新增模块按编号递增，原模块ID保持不变
5. **生成新版PRD**：使用python-docx生成完整docx，保留不变模块的原内容
6. **附对比附录**：在文末添加旧版→新版对比总表
