# 社区电商平台PRD项目 — 虚构案例参考

> 本文档为一个虚构的社区电商平台PRD项目案例，所有内容均为示例用途。
> 项目规模：3个子系统、~15模块、3期交付。
> 展示多子系统PRD的拆解模式、模块编号规范、分批交付策略，可供新项目参考。

## 项目概述

**项目类型：** 社区电商平台
**子系统划分：** 3个子系统（用户端 + 商家端 + 管理后台）

| 子系统 | 模块数 | 说明 |
|--------|-------|------|
| 用户端 | 5模块 | 首页/商品/购物车/订单/个人中心 |
| 商家端 | 5模块 | 商品管理/订单处理/数据看板/结算/客服 |
| 管理后台 | 5模块 | 用户管理/审核管理/运营工具/系统设置/报表 |

## 模块编号规范（可复用）

```
{SubsystemCode}-M{NN}     主模块
{SubsystemCode}-M{NN}-{SS} 子功能点

UC = User Client（用户端）
MC = Merchant Center（商家端）
AD = Admin Dashboard（管理后台）
```

## 用户角色设计

| 角色 | 所属子系统 | 核心操作 |
|:-----|:----------|:--------|
| 普通用户 | 用户端 | 浏览/加购/下单/支付/评价 |
| 商家 | 商家端 | 上架/发货/对账/售后 |
| 运营 | 管理后台 | 审核/活动/推送/数据分析 |
| 超级管理员 | 管理后台 | 全部权限+系统配置 |

## 分批交付模式

| 阶段 | 范围 | 模块数 | 典型内容 |
|:-----|:-----|:-------|:---------|
| 第一期 MVP | 用户端核心+商家端基础 | 8 | 商品浏览/下单支付/商家入驻/订单管理 |
| 第二期 增强 | 商家端完善+管理后台 | 7 | 数据看板/客服/审核/系统设置 |
| 第三期 优化 | 高级功能 | 3+ | 推荐/营销工具/高级报表 |

## 每模块PRD规格模板

### UC-M01 首页

**功能ID：** UC-M01
**优先级：** P0
**页面数：** 3页（首页/搜索页/分类页）

| 功能点 | 操作类型 | 说明 |
|--------|---------|------|
| 商品列表 | 瀑布流展示 | 按推荐算法排序，支持无限滚动 |
| 搜索 | Search | 关键词+分类筛选+排序 |
| 分类导航 | Navigation | 二级分类树形导航 |
| 商品点击 | Navigation | 点击进入商品详情页（UC-M02） |

**数据实体：** Product（商品）, Category（分类）
**Mock数据：** ≥10条商品，含图文/价格/销量/评价数

### MC-M01 商品管理

**功能ID：** MC-M01
**优先级：** P0
**页面数：** 2页（商品列表/商品编辑）

| 功能点 | 操作类型 | 说明 |
|--------|---------|------|
| 商品列表 | Table+Search | 搜索/筛选/分页 |
| 新增商品 | Form | 名称/价格/库存/图片/描述 |
| 编辑商品 | Form | 与新增共用表单 |
| 上下架 | Switch | 一键切换状态 |
| 批量操作 | Batch | 批量上下架/删除 |

## 增量修订工作流

当需求变更或新增政策时，PRD需对齐更新：

**错误做法：** 重新生成文档 → 丢失原有功能点表格和流程图。
**正确做法：** 
1. 索引原文档
2. python-docx精准替换变更内容
3. lxml结构插入新增模块
4. 保留所有原始流程图和功能点表格

## 常见多子系统PRD坑位

| # | 坑位 | 预防 |
|---|------|------|
| 1 | 子系统间字段不一致（用户ID在不同子系统叫法不同） | 统一数据字典 |
| 2 | 模块编号重复 | 按 {子系统}-M{NN} 严格递增 |
| 3 | 跨页面数据流不闭环 | 每个流程标注上下游依赖 |
| 4 | 角色权限遗漏 | 每页标注角色可见性 |
| 5 | Mock数据各子系统不关联 | 统一数据池管理 |
