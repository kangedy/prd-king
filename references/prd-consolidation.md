# PRD文档整合工作流

当用户说"整合成一版本完整PRD"、"全部合并成一份"、"把所有文档合成一个"时使用。

## 触发

- "整合成一版本完整PRD"
- "全部合并成一份"
- "把这些文档合成一个最终版"
- "小乔全部整合成一版本完整PRD"

## 工作流

### Step 1: 确认源文件

列出所有需要整合的源文件，确认路径和内容概要：

```bash
ls -la ~/项目目录/*.md
```

典型场景：多轮迭代后积累的多个版本PRD文件：
- V1.0（初版架构+表格列）
- V2.0（补充章节）
- V3.0（完整表单字段）
- 业务流程梳理文档

### Step 2: 委派子Agent整合

使用 delegate_task 指派产品专家子Agent执行合并：

```
delegate_task(
  goal="整合四份文档为一份完整的XX系统PRD",
  context="""你就是小乔（产品专家）。
  
源文件路径：
1. /path/to/PRD_V1.0.md — 主架构+模块表格列
2. /path/to/PRD_V2.0.md — 补充章节
3. /path/to/PRD_V3.0.md — 完整表单字段
4. /path/to/业务流程.md — 业务流程

用 read_file 读取所有源文件，然后整合为一份完整PRD，用 write_file 输出到 /path/to/PRD_完整版_FINAL.md

按以下结构组织：
1. 文档头(版本/日期/编制)
2. 系统概述与架构
3. 用户角色定义
4. 业务流程全景
5. 逐模块详述(含完整表格列和表单字段)
6. 报表中心
7. 系统管理
8. 关键数据字典
9. 交付批次建议
10. 验收标准

要求：不遗漏任何一个字段，中文输出。
""",
  toolsets=["file"]
)
```

**关键Context要点：**
- 明确告诉子Agent有哪些源文件及路径
- 指定输出文件路径
- 列出目标文档的章节结构
- 强调"不遗漏任何一个字段"
- 指定目标技能（小乔/产品专家）

### Step 3: 验证输出

子Agent完成后验证：
- 文件是否成功写入
- 行数/字节数是否合理（合并后应大于任一源文件）
- 关键章节是否存在（head验证）
- 关键字段是否完整（search_files验证）

```bash
wc -l /path/to/PRD_完整版_FINAL.md
head -20 /path/to/PRD_完整版_FINAL.md
```

### Step 4: 归档旧版本

按照文件整理规范，将旧版本移入 `7_旧版本/` 目录：

```bash
mkdir -p 7_旧版本
mv PRD_V1.0.md PRD_V2.0.md PRD_V3.0.md 7_旧版本/
```

**只保留一份最终版PRD在主目录下。**

## 本章节结构模板

整合后的PRD应包含以下章节：

| 章 | 内容 |
|:--|:-----|
| 1 | 文档头 — 版本/日期/编制/整合来源 |
| 2 | 系统概述 — 产品定位/四层架构图/技术栈+选型建议 |
| 3 | 角色定义 — 所有角色 + 权限矩阵(行=角色, 列=模块) |
| 4 | 业务流程 — 全景图 + 10核心流程 + 状态机 + 核心公式 |
| 5 | 模块详述 — 每个模块: 路径/优先级/表格列/表单字段/搜索/功能点/验收 |
| 6 | 报表中心 — 所有报表列表 + 字段 + 筛选维度 |
| 7 | 系统管理 — 用户/角色/菜单/部门/岗位/字典/参数/日志 |
| 8 | 数据字典 — 所有枚举值 |
| 9 | 交付计划 — 分期/模块/工期 |
| 10 | 验收标准 — 可测量的完成定义 |

## 🚨 致命坑：子Agent压缩陷阱

**这是PRD整合中最严重的错误模式。** 当使用 delegate_task 委派子Agent合并多份PRD时，子Agent几乎**必然**会压缩详细内容。

**典型失败模式：**
- 源V5.0 PRD有1962行，包含83字段车辆信息完整表单、50字段结算导出等详细规格
- 子Agent输出1269行，后管详细设计被压缩成"V4.0保留：83字段完整保留"一句话
- **结果：丢失了所有字段级细节**，只剩摘要骨架

**为什么会发生：** 子Agent没有"不压缩"的意识。它的自然倾向是用摘要替代细节，尤其是当它认为内容"重复"时。

**正确做法：不要委托子Agent做细节合并。** 用终端 `cat` 直接拼接文件：

```bash
cat header.md "小程序PRD.md" separator.md "后管PRD.md" > final.md
```

这保证零删减、零压缩。

## 致命坑：read_file对大文件静默失败

`read_file` 在文件超过 ~96KB 时返回 0 chars / 1 line，**不报错、不提示**。这导致基于 read_file 的文件合并产生空输出。

**症状：** read_file 报告 `content: ''` 或 `1 line`，但文件实际有几千行。

**修复：** 对 >80KB 的文件，用 `terminal cat` 代替 read_file：

```bash
wc -l /path/to/large_file.md  # 先确认行数
cat /path/to/large_file.md > /tmp/output  # 直接操作
```

## 常见坑

1. **子Agent不能访问外部URL** — 如果合并需要来自外部URL的数据，先在主Agent中抓取并保存为本地文件，再传给子Agent
2. **字段覆盖验证** — 合并后行数应 ≥ 各源文件行数之和。如果合并后行数比任一源文件还少，说明发生了压缩。必须用 terminal cat 重做。
3. **版本标记** — 整合后标记为"V1.0 最终交付版"，源文件清单标注行数和来源
4. **子Agent压缩** — 永远不要委托子Agent做细节合并。子Agent适合做架构总览、状态映射、API定义等**新增的整合材料**，不适合做现有文档的机械拼接。机械拼接用 terminal cat。
5. **文件过大时的策略** — 如果合并后超过200KB，考虑分成"主文档（架构+整合）+ 附件A（小程序全文）+ 附件B（后管全文）"的三文件结构
